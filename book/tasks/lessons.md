# CNCbook 작업 Lessons

## 2026-06-18

- CNCbook처럼 원천 글감이 계속 바뀌는 작업은 원천 파일을 직접 덮어쓰기보다 manifest 기반 생성 산출물(`cnc_book_manuscript/`, `brunch/posts/`)로 분리한다.
- macOS 한글 파일명은 조합형/분해형 차이로 manifest 비교가 흔들릴 수 있으므로 NFC 정규화 비교를 기본으로 둔다.
- 브런치 복붙용 파일은 책 원고와 다르게 상단에 제목, 부제, 매거진, 업로드 메모를 명시해야 업로드 직전 실수가 줄어든다.

## 2026-06-23

- manifest의 원천 디렉터리가 바뀌면 기존 생성 파일이 섞이지 않도록 `build` 시작 시 산출물 폴더를 먼저 정리해야 한다.
- 원천 Markdown의 이미지 주석/코드펜스는 generated manuscript 기준으로 다시 확인해야 출판 리스크를 정확히 볼 수 있다.
- 기본 `python3`에 PyYAML이 없을 수 있으므로, 단순 manifest는 fallback parser를 두면 마감형 작업에서 덜 막힌다.
- Word PDF 자동화는 AppleScript/앱 상태에 흔들릴 수 있으므로 XeLaTeX 기반 PDF fallback을 유지한다.
- 책 출력물 용량을 줄일 때는 원본 삽화를 직접 변환하지 말고 `output/` 아래 출력용 JPEG 캐시를 만들어 원고가 그 경로를 참조하게 한다.

## 2026-07-05

- 배포 사이트와 원고 저장소를 합칠 때는 사이트 루트는 배포용으로 유지하고, 제작 소스는 `book/` 같은 하위 폴더로 격리하면 GitHub Pages가 덜 흔들린다.
- 흡수 대상은 원천, 변환 스크립트, 생성 원고, 공개 산출물로 한정하고, 옛 프로젝트 산출물이나 Git 캐시는 가져오지 않는다.
- 레포 이전 뒤에는 GitHub 소유자뿐 아니라 로컬 `origin`, README, 배포 문서의 Pages URL까지 같이 정리해야 오래된 링크가 남지 않는다.

## 2026-07-20

- 원고 내부의 `---`는 장면 구분선이지 페이지 넘김이 아니다. 강제 넘김은 표지, 파트, 장 경계처럼 책 구조가 요구하는 곳에만 둔다.
- 이미지와 다음 캡션을 Pandoc `Figure`로 묶으면 두 요소가 서로 다른 페이지로 갈라지는 문제를 구조적으로 막을 수 있다.
- CJK 글꼴은 PDF 생성 성공만으로 충분하지 않다. Poppler 같은 별도 렌더러에서도 본문이 보이는지 확인하고, 설치형 OTF보다 명시적으로 임베드한 TTF를 우선한다.
- 조판 변경 뒤에는 일부 페이지만 보지 말고 전체 PDF를 이미지로 렌더해 빈 페이지, 넘침, 고아 캡션, 쪽번호 누락을 훑는다.
- Cloudflare Pages의 파일당 25MiB 제한을 배포 전에 확인한다. A5 본문 폭에서는 긴 변 1500px도 300dpi 이상이므로, 삽화 품질을 유지하면서 배포 용량을 줄일 수 있다.
